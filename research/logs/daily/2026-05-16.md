- Task: Research Sunshine and Moonlight architecture and document lessons for Qubox.
- Status: Completed.
- Focus: transport reliability, frame transport, capture/encode pipeline, input model, ecosystem alternatives, and multi-platform design gaps.

- Findings:
	- Qubox's primary weaknesses are packet transport semantics, receive-side buffering and recovery, viewer runtime, and input scope.
	- Sunshine and Moonlight are reliable because they combine packetization, jitter handling, FEC, session-local worker isolation, and richer input plus scaling models.
	- Rust is mature enough to own the architecture, control plane, transport orchestration, input semantics, and large parts of audio and windowing, but hardware video and some platform IO still need native integrations.

- Upgrade direction:
	- use incremental replacement rather than a big-bang rewrite,
	- prioritize packetized transport and client receive path before viewer polish or deeper capture work,
	- replace the current viewer and input runtime after the transport foundation is corrected.
	- keep Sunshine and Moonlight as methodology references for packetization, receive recovery, input semantics, scaling, and backend strategy, not as literal templates.
