# P2-16: 4K144 Streaming

Status: research complete, implementation pending.
Owner: `apps/host-agent` (encoder), `apps/client-cli` (decoder/display), with a new `presets` module.
Depends on: P0-1 (HW encode), P0-3 (HW decode), P0-2 (datagram media path), P0-5 (frame pacing).
Blockers: requires a real 4K144 setup to test (Xephyr is 1024x768); CI smoke test will use 1080p60 as a proxy.

## Goal

Support 4K (3840×2160) at 144 fps streaming end-to-end. The pipeline is GPU-bound, not network-bound: HW encode is mandatory, HW decode is mandatory, the wire must support ~100-150 Mbps, and the display path needs HDMI 2.1 / DisplayPort 1.4 DSC. The latency budget stays under 30 ms on LAN.

## Research Summary

### Bitrate requirements (4K144)

| Codec | Bitrate (good quality) | HW encode support | Notes |
|-------|------------------------|-------------------|-------|
| H.264 | 150+ Mbps | NVENC, VAAPI, QSV, AMF, VideoToolbox | Most compatible, least efficient |
| H.265 (HEVC) | 100-150 Mbps | NVENC (Maxwell+), VAAPI (KBL+), QSV, AMF, VideoToolbox | Best balance |
| AV1 | 80-120 Mbps | NVENC (Ada+), VAAPI (Arc+ / RDNA2+), QSV (Arc+), AMF (RDNA3+) | Most efficient, highest encode cost |

**Choice: HEVC for compatibility, AV1 for bandwidth-constrained scenarios.** H.264 is the fallback for very old clients.

### Codec latency (encoder, 4K144)

| Encoder | H.264 | H.265 | AV1 |
|---------|-------|-------|-----|
| NVENC | 5-10 ms | 8-12 ms | 10-20 ms (B-frames) |
| VAAPI | 7-12 ms | 10-15 ms | 12-20 ms |
| QSV | 10-15 ms | 12-18 ms | 15-25 ms |
| AMF | 10-15 ms | 12-18 ms | 12-22 ms |
| VideoToolbox | 8-12 ms | 10-15 ms | 12-18 ms (M3+) |
| libx264 (SW) | 30-50 ms | n/a | n/a |
| libx265 (SW) | n/a | 50-100 ms | n/a |
| rav1e / aomenc (SW) | n/a | n/a | 100+ ms (impossible) |

**Conclusion: HW encode is mandatory for 4K144.** Software encoders cannot keep up.

### Codec latency (decoder, 4K144)

| Decoder | H.264 | H.265 | AV1 |
|---------|-------|-------|-----|
| NVDEC | 1-2 ms | 2-3 ms | 2-4 ms |
| VAAPI | 1-2 ms | 2-3 ms | 2-4 ms |
| D3D11VA | 1-3 ms | 2-4 ms | 2-5 ms |
| VideoToolbox | 2-3 ms | 2-3 ms | 3-4 ms |
| libx264 (SW) | 8-15 ms | n/a | n/a |
| libx265 (SW) | n/a | 15-30 ms | n/a |
| dav1d (SW) | n/a | n/a | 20-30 ms |

**HW decode is mandatory for 4K144.**

### Capture (4K144)

- DXGI Desktop Duplication: 1-3 ms. GPU memory bandwidth is the bottleneck (~14 GB/s for 4K144 BGRA).
- Wayland wlr-screencopy: 2-5 ms.
- ScreenCaptureKit: 5-15 ms.
- X11 x11grab: 5-10 ms (slower; CPU-bound).

**DXGI is the best path for 4K144 on Windows.**

### Wire bandwidth

- 4K144 at 120 Mbps + 20% FEC = 144 Mbps wire. 1 Gbps Ethernet handles this trivially.
- QUIC datagrams: 1200-byte MTU → ~15,000 packets/sec. The kernel UDP stack and the network card handle this on 1 Gbps.
- On WAN: most residential links cap at 100-300 Mbps upload. 4K144 over WAN is rare but possible with a 500+ Mbps link.

### Display

- wgpu `Mailbox` at 4K144: frame-pacing latency = 1 vblank = 6.9 ms (at 144 Hz).
- Texture upload: 4K RGBA = 33 MB; at 144 fps = 4.7 GB/s. Modern GPUs handle this.
- HDMI 2.1 / DisplayPort 1.4 DSC: required for 4K144. HDMI 2.0 caps at 4K60.

### End-to-end latency (LAN)

| Stage | Latency |
|-------|---------|
| Capture (DXGI) | 2 ms |
| Encode (NVENC HEVC) | 10 ms |
| Wire (QUIC datagrams) | 5 ms |
| Decode (NVDEC) | 2 ms |
| Upload (wgpu write_texture) | 1 ms |
| Vblank (144 Hz) | 7 ms |
| **Total** | **27 ms** |

Within budget for non-competitive gaming. Competitive gamers at 4K144 on a 1 Gbps LAN will see ~27 ms glass-to-glass.

### End-to-end latency (WAN, 50 ms wire)

| Stage | Latency |
|-------|---------|
| Capture + encode | 12 ms |
| Wire (50 ms) | 50 ms |
| Decode + present | 10 ms |
| **Total** | **72 ms** |

Within budget for casual gaming over WAN.

### GPU requirements

**Host**:
- NVIDIA: RTX 30+ for HEVC, RTX 40+ for AV1. AV1 B-frames limited to ~120 fps on RTX 40 (not 144); for 144 fps AV1, wait for RTX 50.
- Intel: Arc+ for AV1 HEVC; older Intel iGPUs do HEVC only.
- AMD: RX 7000+ for AV1.
- Apple: M3+ for AV1 (M1/M2 do HEVC only).

**Client**:
- Any modern GPU with HW decode for the chosen codec.
- Display: 4K144 monitor via HDMI 2.1 or DisplayPort 1.4 DSC.

### Steam Deck OLED context

- 1280×800, 90 Hz, HDR-capable. Not 4K144, but the same encoder/decoder pipeline works.
- Steam Deck 2 (rumored 2026) may bump to 1080p120 or 1440p120.

### Bitrate calculation

```rust
pub fn recommended_bitrate(width: u32, height: u32, fps: u32, codec: VideoCodec) -> u32 {
    let pixels = width * height * fps;
    // 0.15 bits per pixel per second for H.264, 0.10 for HEVC, 0.07 for AV1
    let bpp = match codec {
        VideoCodec::H264 => 0.15,
        VideoCodec::Hevc => 0.10,
        VideoCodec::Av1  => 0.07,
    };
    ((pixels as f32) * bpp) as u32
}

assert_eq!(recommended_bitrate(3840, 2160, 144, VideoCodec::Hevc), 119_439_360);  // ~119 Mbps
assert_eq!(recommended_bitrate(3840, 2160, 144, VideoCodec::Av1), 83_607_552);    // ~84 Mbps
```

### 2024-2026 status

- **NVENC AV1 on RTX 40+**: mature for 60-120 fps at 4K. 144 fps requires B-frames off; AV1 B-frames are limited.
- **HEVC on NVENC / VAAPI / QSV / AMF**: mature, 4K144 supported across the board.
- **AV1 SW decode (dav1d)**: 20-30 ms at 4K144, not usable. HW decode is mandatory.
- **DisplayPort 1.4 DSC**: standard on 4K144 monitors; required for HDMI 2.0 → 4K60.
- **4K144 monitors**: $300+; standard in 2024-2026.
- **Cloud 4K144**: GeForce Now and Xbox Cloud Gaming offer 4K60 but not 4K144.

## Implementation Plan

### Step 1: 4K144 preset

`apps/host-agent/src/encoder/presets.rs` (new):
- `pub fn preset_4k144(codec: VideoCodec) -> EncoderConfig`:
  - codec-specific encoder selection (no H.264; HEVC or AV1).
  - bitrate from the `recommended_bitrate` function.
  - GOP = 240 (4K144 has more keyframe budget; same recovery).
  - B-frames off (latency).
  - preset p1 / veryfast / speed.
  - CBR.

### Step 2: Encoder selection

`apps/host-agent/src/encoder/pipeline.rs` (existing, extended):
- For 4K144, prefer HEVC over AV1 unless the user picks AV1 explicitly.
- For 4K60 and below, all codecs are options; H.264 for compatibility, HEVC/AV1 for bandwidth.
- The `--codec {h264,hevc,av1}` CLI flag and the `VideoStreamPreferences.codec` field.

### Step 3: Decode pipeline

`apps/client-cli/src/decoder/ffnext.rs` (P0-3):
- Same path as 4K60, but the ffmpeg-next decoder must support the higher frame rate.
- Verify NVENC NVENC AV1 handles 144 fps; if not, fall back to HEVC.
- The decoder's `avcodec_send_packet` / `avcodec_receive_frame` loop is identical; the only difference is more iterations per second.

### Step 4: Frame pacing

`apps/client-cli/src/frame_pacing/pacer.rs` (P0-5):
- Refresh rate detection for 144 Hz displays.
- vblank interval = 6.94 ms.
- The pacer is identical; just a different interval.

### Step 5: Wire format

The existing `WireAccessUnitHeader` carries the width, height, fps, and codec. No new fields. The chunking in P0-2 already handles large frames (a 4K HEVC frame is ~150 KB at 120 Mbps / 144 fps; ~125 chunks of 1200 bytes each).

### Step 6: Memory pressure

4K144 capture pipeline uses ~1 GB of GPU memory (texture + staging buffer + decoder surfaces). Plan for 2-3 GB per stream. On multi-monitor (P1-7) with 2 displays at 4K144, this is 4-6 GB. Modern GPUs have 8-16 GB; we're fine.

### Step 7: Tests

- Unit test: `recommended_bitrate` returns expected values for 4K144.
- Integration test on Xephyr (1080p60 proxy): verify the encoder/decoder/wire/pacer works at high bitrate; the Xephyr test is a smoke test, not a true 4K144 test.
- Manual: 4K144 host + 4K144 client on a 1 Gbps LAN, verify the wire doesn't drop packets and the latency is ~30 ms.
- Latency test: capture-to-display < 30 ms on LAN.

## Risks and Open Questions

- **NVENC AV1 4K144 B-frames**: NVENC AV1 on Ada (RTX 40) supports 4K120 with B-frames; 4K144 requires either no B-frames or wait for Blackwell (RTX 50). For 4K144, HEVC is the safer choice.
- **DisplayPort 1.4 DSC vs HDMI 2.1**: DisplayPort 1.4 DSC is standard on 4K144 monitors; HDMI 2.1 is required for HDMI. Verify the user's cable.
- **CPU usage on the host**: 4K144 capture + HW encode uses ~5% CPU. Plenty of headroom for the audio pipeline.
- **CPU usage on the client**: 4K144 decode + upload + wgpu blit uses ~5-10% CPU. Plenty of headroom.
- **Wire bandwidth on residential WAN**: most users have 100-300 Mbps upload. 4K144 at 120 Mbps is achievable on a 200+ Mbps link; below that, 4K60 or 1440p120 is more practical.
- **Network card**: a 1 Gbps NIC is required; 100 Mbps won't keep up. Most modern devices have 1 Gbps Ethernet or Wi-Fi 6 (~700 Mbps real).
- **FEC overhead at high bitrate**: 20% FEC on 120 Mbps = 24 Mbps overhead. On a 200 Mbps link, the user has 56 Mbps for everything else (browsing, etc.). Plan for users to disable FEC for 4K144.
- **GPU memory fragmentation**: 4K144 capture creates 33 MB textures; allocate from a pool to avoid fragmentation.
- **AV1 SW decode on the client**: dav1d is 20-30 ms at 4K144; some clients (Steam Deck Linux) may not have HW AV1 decode. Plan a fallback to HEVC.
- **DisplayPort 1.4 DSC artifacts**: DSC is visually lossless; some games show subtle banding in dark scenes. Disable DSC if the user's GPU + monitor support 4K144 native (rare; only HDMI 2.1 4K144 native exists on a few high-end monitors).
- **The dev box (Xephyr 1024×768)**: cannot test 4K144. Manual testing on real hardware required.
